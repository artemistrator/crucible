# AI Orchestrator

**AI-система управления проектами и автоматизации разработки, превращающая идеи в структурированные планы.**

AI Orchestrator декомпозирует идеи в задачи, управляет проектами с помощью AI-агентов, использует RAG для управления контекстом, предоставляет возможности веб-поиска и обучается на прошлых проектах. Включает режим автоматического выполнения задач.

## Технический паспорт (v1.0 Demo)

Этот раздел описывает рефакторинг v1.0: **надёжное QA (Hard Evidence), изоляция Docker DooD, pgvector RAG, OpenVDL (vibe.yaml), RCA-ретраи, правила AHP-диспетчера и защитные механизмы.** Используйте как основной контекст для дальнейшей разработки.

### 1. Надёжное QA (Hard Evidence)

- **Механизм**: QA больше не полагается на хрупкое совпадение строк ("PASS", "✓") в отчётах. Доказательства структурированы и проверяются явно.
- **На стороне исполнителя** (`lib/agents/task-executor-agent.ts`, `lib/agents/execution-agent.ts`): Перед отправкой отчёта в QA, исполнитель запускает **шаги верификации** из `verificationCriteria`:
  - Для каждого пути в `artifacts` выполняет `ls -la <path>` и `cat <path>` (или аналог) и добавляет вывод в отчёт.
  - Запускает команду `automatedCheck` и добавляет stdout/stderr.
  - Вывод «Artifacts Check» и «automatedCheck» встраивается в отчёт, предоставляя QA **конкретные доказательства** (наличие файла, коды завершения команд, логи).
- **На стороне QA** (`lib/agents/qa.ts`): Двухфазная верификация:
  - **Жёсткие ворота (код)**: Критерии верификации считываются из задачи (`artifacts`, `automatedCheck`, `manualCheck`). Промпт инструктирует модель искать **доказательства для каждого критерия** (например, вывод `ls -la`, логи тестов, успех сборки). Отсутствие доказательств → REJECT с сообщением «Missing evidence for: ...».
  - **Мягкие ворота (LLM)**: Семантическая проверка кода и рассуждений; JSON-ответ парсится через `extractJsonFromText` (блоки с кавычками или первый `{` до последнего `}`), затем валидируется по `qaVerificationSchema` (статус APPROVED/REJECTED, reasoning, confidence). REJECT с низкой уверенностью может вызвать запрос второго мнения.

### 2. Изоляция и безопасность (Docker DooD)

- **Механизм**: **Docker-out-of-Docker (DooD)**. Каждая сессия выполнения запускается в собственном **временном контейнере**; код проекта на хосте монтируется через bind-mount.
- **Реализация** (`lib/execution/container-manager.ts`):
  - `ensureContainer(sessionId, projectPath)` — создаёт долгоживущий контейнер (например, `node:20-slim`) с именем `ai-orch-session-<sessionId>`, монтированием `hostProjectPath:/app/project` и пользователем `HOST_UID:HOST_GID` (файлы принадлежат хост-пользователю, без проблем с root).
  - `executeInContainer(sessionId, command, timeoutMs)` — запускает `docker exec` с **таймаутом по умолчанию 5 минут**; при истечении выбрасывает исключение; возвращает `{ stdout, stderr, exitCode }`.
  - `destroyContainer(sessionId)` — выполняет `docker rm -f` контейнера сессии.
  - Пути хоста разрешаются через `HOST_PROJECTS_DIR`; внутренние пути вида `/app/projects/...` транслируются в `HOST_PROJECTS_DIR/...`.
- **Окружение**: `HOST_PROJECTS_DIR`, `HOST_UID`, `HOST_GID` читаются из `.env` для переносимости (Mac/Linux).
- **Сборка мусора** (`instrumentation.ts`): При запуске Next.js (Node runtime) вызывается `cleanupOrphanedContainers()`: находит все контейнеры с именами, начинающимися на `ai-orch-session-`, и удаляет их, чтобы старые сессии не оставляли устаревших контейнеров.

### 3. Быстрый RAG (pgvector)

- **Оптимизация**: Вычисление похожести выполняется **внутри PostgreSQL** через pgvector (оператор `<=>`), а не в JavaScript-циклах.
- **API** (`lib/rag/search.ts`):
  - `searchGlobalInsights(query, limit)` — семантический поиск по `GlobalInsight` (исторические уроки); используется decompose-idea и generate-tasks для инъекции прошлых инсайтов в промпты.
  - `searchSimilar` / поиск в рамках проекта — используется инструментом **searchCodebase**.
- **Инструменты агентов** (`lib/agents/tools.ts`):
  - **searchCodebase** (`createSearchCodebaseTool`): Встраивает запрос, выполняет raw SQL-запрос по `FileEmbedding` с JOIN к `ProjectFile` (фильтр `similarity > 0.4`), возвращает `{ filePath, content, similarity }`.
  - **getCodeMap** (`createGetCodeMapTool`): Возвращает **структурную карту** проекта: пути файлов + сущности кода (классы, функции и т.д.) из таблицы `CodeEntity` — без тел файлов, чтобы агент видел «скелет» кодовой базы без чтения полных файлов.

### 4. OpenVDL (vibe.yaml)

- **Механизм**: **«Генетический код»** проекта. Агенты автоматически получают правила из конфигурационного файла в корне проекта.
- **Реализация**:
  - **Парсер** (`lib/vibe/parser.ts`): `loadProjectVibe(projectId)` читает `vibe.yaml` или `vibe.json` из директории проекта (через `getProjectDir`), парсит через `yaml.parse` или `JSON.parse`; возвращает `VibeConfig | null`; никогда не выбрасывает исключение (отсутствующий файл → null).
  - **Схема** (`lib/vibe/types.ts`): `VibeConfig` включает `architecture` (preferred_pattern, forbidden_patterns), `code_style` (naming, error_handling), `testing` (framework, require_for), `qa_rules` (mandatory_evidence, strict_guidelines).
  - **Инъекция** (`lib/agents/prompt-generator.ts`): При генерации промпта задачи вызывается `loadProjectVibe(project.id)`; при наличии файла строится **секция vibe** (предпочтительный паттерн, запрещённые паттерны, именование, обработка ошибок, фреймворк тестирования, правила QA) и добавляется к системному промпту, чтобы исполнитель следовал архитектуре и стандартам QA без лишних инструкций в тексте задачи.

### 5. Умные ретраи (RCA)

- **Механизм**: **Root Cause Analysis (RCA)**. При выполнении тикета (fix-after-reject) исполнитель **сначала** должен вывести шаг-мысль с объяснением причины предыдущего провала и описанием изменений.
- **Реализация** (`lib/agents/task-executor-agent.ts`): Для `TICKET_REQUEST` системный промпт содержит строгое правило: **первый** элемент массива `steps` **обязан** быть **мысленным шагом** (без `toolName`) с RCA. Промпт указывает: «ПЕРВЫМ элементом в массиве `steps` ДОЛЖЕН быть мысленный шаг... с Root Cause Analysis (RCA) предыдущей неудачной попытки» и «НЕ используйте инструменты до вывода этой мысли RCA». Агент логирует эти мысли, что снижает повторение одинаковых ошибок.

### 6. Стабильность AHP-диспетчера

- **Зависимости** (`lib/execution/agent-factory.ts`, `lib/agents/team-lead-agent.ts`): Следующая доступная для выполнения задача выбирается через Prisma: `dependencies: { every: { dependsOn: { status: "DONE" } } }`. Задача запускается только когда **все** её `dependsOn`-задачи имеют статус DONE.
- **Строгое завершение** (`app/api/execution-sessions/run-ahp/route.ts`, `lib/execution/agent-factory.ts`): Сессия считается завершённой только когда **все** задачи плана имеют статус DONE (и нет открытых тикетов). `isSessionComplete()` проверяет `planTasks.every(t => t.status === "DONE")` и отсутствие ожидающих сообщений.
- **Надёжный парсинг**: В task executor ответ LLM нормализуется с помощью регулярного выражения для извлечения единственного JSON-объекта: `planJsonText.replace(/^[\s\S]*?(\{[\s\S]*\})[\s\S]*$/m, "$1")`. При ошибке парсинга или валидации тикет устанавливается в REJECTED и создаётся **системный комментарий** с сообщением об ошибке (`createTicketRejectionComment`), чтобы выполнение не зависало и пользователь видел причину.

### 7. Защитные механизмы

- **Таймауты**: Каждая команда в контейнере сессии использует `executeInContainer(..., timeoutMs)` с дефолтом **5 минут** (`DEFAULT_COMMAND_TIMEOUT_MS` в `container-manager.ts`). По истечении таймаута выбрасывается `AbortError`, результат команды считается провальным.
- **Сборка мусора**: См. §2 — `cleanupOrphanedContainers()` в `instrumentation.ts` при запуске приложения.

## Основные возможности

### 1. AI-декомпозиция

- **Идея → Планы**: Введите идею на естественном языке, получите 3 архитектурных плана с разными технологическими стеками
- **План → Задачи**: Автоматическая декомпозиция с зависимостями и критериями верификации
- **Глобальная память**: AI обучается на удалённых проектах и применяет инсайты к новым проектам
- **Умная оценка сложности**: Автоматическая оценка размера проекта (S/M/L/XL)

### 2. Умное управление задачами

- **Канбан-доска**: TODO → IN_PROGRESS → REVIEW → WAITING_APPROVAL → DONE
- **Human-in-the-Loop**: Опциональный шлюз подтверждения перед переходом в DONE
- **Назначение агентов**: Frontend, Backend, DevOps, Teamlead, Cursor, QA-агенты
- **Граф зависимостей**: Визуализация и управление зависимостями задач через @xyflow/react
- **Детальные промпты**: AI-генерация промптов для кодинга для каждой задачи

### 3. Режим автоматического выполнения

- **Полная автоматизация**: Execution agent берёт задачи и выполняет их автоматически
- **Консоль в реальном времени**: Живой терминал с выполнением команд и ответами AI
- **Контроль затрат**: Установка лимитов расходов на сессии выполнения
- **Пауза/Возобновление**: Контроль выполнения в любой момент
- **Подтверждение команд**: Опциональное ручное подтверждение каждой команды

### 4. QA Code Review

- **Веб-поиск**: Интеграция Tavily API для актуальной документации
- **Чтение файлов**: QA-агент читает файлы проекта через инструмент `readFile`
- **Строгая верификация**: Отклонение задач без конкретных доказательств (логи тестов, вывод сборки)
- **Анализ ошибок**: Автоматический анализ сбоев и конкретная обратная связь

### 5. RAG и контекст

- **Синхронизация файлов**: Синхронизация файлов в реальном времени с RAG-эмбеддингами
- **Векторный поиск**: Семантический поиск по файлам проекта на базе pgvector
- **Сущности кода**: Автоматическое извлечение классов, функций, импортов
- **Глобальный контекст**: Проектные правила и контекст для всех агентов

### 6. Динамическое перепланирование

- **Агент-архитектор**: Обновление оставшихся задач на основе выполненной работы
- **Анализ влияния**: Автоматическая корректировка зависимых задач
- **Генерация ADR**: Architecture Decision Records для изменений плана

### 7. FinOps — учёт затрат

- **Автоматический трекинг**: Все вызовы AI логируются с токенами и стоимостью
- **Дашборд проекта**: Детализация затрат по проекту
- **Глобальный биллинг**: `/api/billing` для общих расходов
- **Сравнение моделей**: Отслеживание затрат между разными AI-провайдерами

### 8. CLI-инструмент

Командная строка для интеграции внешних проектов:

```bash
# Инициализация подключения
orchestrator init

# Получить следующую задачу
orchestrator task next

# Отчёт о прогрессе
orchestrator report <taskId> --content "Работа выполнена..." --file path/to/file.ts

# Отметить задачу выполненной
orchestrator done <taskId>

# Получить AI-промпт для задачи
orchestrator prompt <taskId>
```

## Установка и настройка

### 1. Клонирование репозитория

```bash
git clone <your-repo-url>
cd ai-orchestrator
```

### 2. Настройка окружения

```bash
cp .env.example .env
```

**Обязательные переменные:**

| Переменная | Описание | Обязательно |
|------------|----------|-------------|
| `DATABASE_URL` | URL PostgreSQL с pgvector | ✅ |
| `OPENAI_API_KEY` | Ключ OpenAI API | Для OpenAI |
| `ANTHROPIC_API_KEY` | Ключ Anthropic | Для Claude |
| `ZAI_API_KEY` | Ключ Z.ai (GLM) | Для Z.ai |
| `TAVILY_API_KEY` | Ключ Tavily | Для веб-поиска |

### 3. Развёртывание через Docker

```bash
# Первый запуск (чистая база)
docker compose down -v
docker compose up -d --build

# Последующие запуски
docker compose up -d --build
```

**Сервисы:**
- Приложение: http://localhost:3002
- Adminer (DB UI): http://localhost:8081
- PostgreSQL: localhost:5433

### 4. Миграции базы данных

Миграции выполняются автоматически при первом запуске. Для ручного обновления:

```bash
docker compose exec app npx prisma db push
```

### 5. Режимы sync-клиента и окружение

- **Sync-клиент на хосте (рекомендуется для локальной разработки):**
  - Запустите `node sync-client.js` из скачанного кита на хост-машине.
  - По умолчанию подключается к `http://localhost:3002/api/sync` (экспортируется из docker).
  - Можно переопределить через `--url http://localhost:3002/api/sync`.
- **Sync-клиент на сервере (внутри контейнера, автозапуск):**
  - Управляется флагами окружения:
    - `SYNC_CLIENT_AUTOSTART=true` — разрешить серверу запускать процессы sync-client для каждого проекта.
    - `NEXT_PUBLIC_SYNC_CLIENT_AUTOSTART=true` — подсказка UI о том, что sync-client запущен на стороне сервера.
  - Внутри docker приложение слушает `http://app:3000`; `INTERNAL_APP_URL` задаётся соответственно.

## Автоматическое выполнение

Запуск автоматического выполнения задач из веб-интерфейса:

1. Откройте страницу плана проекта
2. Нажмите кнопку «Start Execution»
3. Настройте:
   - **Auto-approve**: пропуск подтверждений команд
   - **Cost limit**: максимальный бюджет сессии
4. Следите за прогрессом в Execution Console

Execution agent будет:
1. Брать ожидающие задачи
2. Генерировать детальные промпты
3. Выполнять команды
4. Верифицировать завершение через QA
5. Переходить к следующей задаче

## Семантический поиск по коду (`searchCodebase`)

AI Orchestrator ведёт семантический индекс синхронизированных файлов проекта:

- **Sync-клиент** (`public/sync-client.js` или `.orchestrator/sync-client.js` в проекте) отслеживает рабочее пространство и отправляет обновления файлов в API.
- Эндпоинт `/api/sync` хранит/обновляет записи `ProjectFile` и запускает RAG-пайплайн (`processFile`) для создания/обновления строк `FileEmbedding` в PostgreSQL с pgvector.
- Каждый файл разбивается на чанки и встраивается; чанки хранятся с привязкой к исходному `ProjectFile`.

Агент TASK_EXECUTOR может вызывать инструмент `searchCodebase` для **семантического поиска по всей кодовой базе**:

- Генерирует эмбеддинг для запроса на естественном языке.
- Выполняет pgvector-запрос по `FileEmbedding` с JOIN к `ProjectFile`, ограниченный текущим `projectId`.
- Возвращает наиболее релевантные чанки как `{ filePath, content, similarity }`, отфильтрованные по `similarity > 0.4` и отсортированные по убыванию похожести.

Требования к работе:

- `DATABASE_URL` должен указывать на PostgreSQL с включённым расширением `pgvector`.
- Таблицы `FileEmbedding` и `GlobalInsight` должны иметь колонку `embedding`, совместимую с pgvector-операторами (`<=>`).
